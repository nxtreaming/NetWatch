## 审计范围

1. **认证与授权**
   - 是否复用主应用登录态/权限控制
   - 未登录/越权访问是否被拒绝
2. **输入校验**
   - `$_GET`/`$_POST` 参数白名单、类型、长度
   - 是否存在宽松比较和未过滤输入
3. **CSRF 防护**
   - 所有写操作是否强制 CSRF
   - 只读接口是否仅允许安全方法
4. **XSS/输出编码**
   - HTML 上下文使用 `htmlspecialchars`
   - JS 上下文使用 `json_encode`
5. **API 响应一致性**
   - 错误码与消息是否规范
   - 异常路径是否泄露敏感细节
6. **文件路径与依赖加载**
   - `require_once` 是否统一使用绝对路径（`__DIR__`）
7. **数据库与并发**
   - SQL 是否全部参数化
   - 并发更新是否有锁等待/错误处理
8. **日志与可观测性**
   - 关键操作是否记录日志
   - 错误日志是否可定位问题

## 执行步骤（建议）

1. 通读 `proxy-status/` 全部入口文件并标注风险点。
2. 按 H/M/L 级别输出问题清单与修复建议。
3. 逐项修复并在每项后记录变更文件与行号。
4. 完成后更新本文档状态为 `Done`，并附修复汇总。

## 交付标准

- 形成独立审计结果（可并入主审计报告）
- 对高风险问题有明确修复证据（文件/行号）
- 子应用入口具备与主应用一致的安全基线
